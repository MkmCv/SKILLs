## NotebookLM 官方 Steering Prompt 能力

根据 [Google 官方文档](https://support.google.com/notebooklm/answer/16454555)，NotebookLM 视频概览的 Steering Prompt 支持：
- **聚焦特定主题**（Focus on specific topics）
- **指定目标受众**（Target a specific audience）  
- **提供额外上下文**（Provide context）

本 Skill 生成的 Prompt 遵循这些官方支持的自定义方向，不超出 NotebookLM 的能力边界。

## Overview

把 **md 文档**转换成三类视频脚本产物：

- **A｜产品/商务 Feature 视频**：不讲实现，只讲变化与价值（面向非技术观众）
- **B｜技术原理讲解视频**：动机→直觉→机制→术语映射（面向有基础的学习者）
- **C｜SDK 版本发布视频**：基于 CHANGELOG.md，按 Added/Changed/Fixed/Deprecated 结构讲解（面向开发者，作为 SDK 产品发布讲解员帮助用户快速了解版本变化）

默认输出（唯一必出）：
- `notebooklm_prompt.md`：**可直接粘贴到 NotebookLM**，引导其基于你上传的 Sources（文档+图片）生成视频讲解

可选输出（默认建议生成，但不强制）：
- `assets_checklist.md`：建议补哪些截图/图片/示意图（P0/P1/P2），由用户与 AI 商榷是否补充

## Trigger Conditions（何时使用）

当用户出现以下意图时使用本 Skill：

- "把文档做成视频/讲解视频/宣传视频/培训视频/NotebookLM 视频"
- "我不想看文档，2 分钟讲清楚"
- "纯文本太痛苦，想要原理讲解 + 图示化"
- 用户提供/引用 `*.md`、release note、PRD、技术设计文档、**CHANGELOG.md / 版本发布说明**等

## Inputs（输入）

至少需要：
- **文档内容**：Markdown 原文或其关键章节

可选输入（建议有）：
- **视频类型**：`A`（产品/商务）、`B`（技术原理）或 `C`（版本发布）
- **目标受众**：例如"产品/商务/老板"或"后端工程师/学生"或"开发者/技术用户"
- **讲解节奏**：
  * `紧凑`：A 产品视频 2-3min、B 技术视频 3-4min、C 版本发布 5-6min
  * `标准`：A 产品视频 4-5min、B 技术视频 5-7min、C 版本发布 7-9min
  * `从容`：A 产品视频 6+min、B 技术视频 8+min、C 版本发布 10+min
- **章节结构**：如果 md 有清晰标题层级，提供会更稳定
- **配图策略**：不配图 / 只补 P0（推荐）/ 尽量多图
- **输出偏好**：只要 `notebooklm_prompt.md`，还是 `assets_checklist.md + notebooklm_prompt.md`（推荐后者）

## Outputs（输出）

### 必出：NotebookLM Prompt（给你一键用）

生成 `notebooklm_prompt.md`，内容为"角色 + 目标 + 结构 + 画面规则 + 禁止项"，并引用 `templates/` 下对应模板。

要求：
- **必须假设**：用户已在 NotebookLM 的 Sources 中上传了文档与（可选）图片/截图
- **不得要求**：用户再粘贴全文或复述全文
- **必须包含**：信息来源约束（不发明）、可读性失败条件（看不清则失败并调整画面）、以及"图片是建议项"的说明

### 可选：素材/截图建议清单（建议生成，但不强制）

生成 `assets_checklist.md`（Markdown 待办清单），内容包括：
- 建议哪些截图/示意图/对比图
- 对于类型 C（版本发布）：建议哪些技术术语需要形象化图示（如 UVC → USB 摄像头图标）
- 每张图用途、截图范围、需要遮挡的信息、优先级（P0/P1/P2）
- 替代方案：没有截图就用 mock/示意图

重要边界：
- 图片/截图是**建议项**，由用户与 AI 商榷是否补、补多少
- 不要把"建议"写成"强制"

## Workflow（强制流程）

0. **先收集视频要求（如果用户没说清楚就必须问）**

当用户没有明确给出以下关键参数中的任意 2 项时，先问一次"快速选择题"，等待用户回答后再继续：

- 视频类型：A（产品/商务）、B（技术原理）或 C（版本发布）
- 讲解节奏：紧凑（2-3min）/ 标准（3-5min）/ 从容（5-6min）
- 目标受众：例如 产品/商务/老板 或 工程师/学生 或 开发者/技术用户
- 配图策略：不配图 / 只补 P0（推荐）/ 尽量多图

可直接复制给用户的问题（一次问完，用户一句话选项即可）：

> 你希望这次视频怎么做？请按下面格式回复：  
> 1) 类型：A（产品/商务）或 B（技术原理）或 C（版本发布）（不确定我也可以推荐）  
> 2) 节奏：紧凑（A 2-3min、B 3-4min、C 5-6min）/ 标准（A 4-5min、B 5-7min、C 7-9min）/ 从容（A 6+min、B 8+min、C 10+min）  
> 3) 受众：产品商务 / 工程师 / 开发者 / 新手 / 自定义  
> 4) 配图：不配图 / 只补P0（推荐） / 尽量多图  
> 5) 输出：只要 `notebooklm_prompt.md`，还是 `assets_checklist.md + notebooklm_prompt.md`（推荐后者）

1. **识别视频类型**
   - 若用户明确 A/B/C：照做
   - 若用户提供 CHANGELOG.md 或版本发布说明：优先推荐 C
   - 若不明确：根据受众/措辞判断；仍不确定则先输出"我建议 X + 原因"

2. **抽取信息（只抽文档内）**
   - A：抽取「新增能力/变化点」「旧方式痛点」「用户视角使用方式」「价值指标（效率/成本/风险/体验）」
   - B：抽取「动机/问题」「核心概念」「流程/因果链」「关键术语定义」「常见误解/边界」
   - C：**从 CHANGELOG 原文抽取**，按以下结构组织：
     * **Added（新增功能）**：识别重要新增（如算法组件、协议库、新驱动），每个重要特性 30-40 秒讲解
     * **Changed（变更）**：API 变更、行为调整、依赖更新等
     * **Fixed（问题修复）**：按模块分类总结（驱动/组件/系统），不逐条展开
     * **Deprecated（废弃）**：如有，提醒用户迁移
     * **技术术语识别**：识别需要形象化图示的术语（如 MQTT → 消息传递图、DMA → 数据搬运示意图、DVP → 摄像头接口图标）
     * **重要**：只使用 CHANGELOG 中的原文描述，不要补充或推测功能细节、性能数据、使用场景
     * **预告约束**：只在 CHANGELOG 明确提及"Roadmap""Next Version""Planned Features"时才生成预告，否则直接说"欢迎升级体验"
     * **角色定位**：作为 SDK 产品发布讲解员，帮助开发者快速了解版本变化，而非深入教学
   - 绝不扩展到文档未覆盖的内容

3. **生成素材建议（可选｜默认建议生成）**
   - 识别哪些镜头更适合 **真实截图**（UI、报表、配置页面等） vs **示意图**（流程/结构/对比）
   - 对于类型 C（版本发布）：识别需要"形象化图示"的技术术语（如 UVC → USB 摄像头图、MQTT → 消息传递图、人脸识别 → 人脸框图标）
   - 输出 `assets_checklist.md`（P0/P1/P2）
   - 若关键截图/图标缺失：在清单中明确向用户说明"建议用什么图示/图标来解释哪个术语"；但不得要求"必须提供"

4. **生成 NotebookLM Prompt（必须包含讲解节奏控制 + 图片可读性硬约束 + 图示内容边界）**
   - A 使用 `templates/template_a_feature.md`
   - B 使用 `templates/template_b_tech.md`
   - C 使用 `templates/template_c_release.md`
   - 公共视觉规则追加 `templates/visual_rules.md`
   - **强制包含讲解节奏控制策略**：
     * 按用户指定节奏（紧凑/标准/从容）明确告知 NotebookLM：
       - 紧凑：短句、快速过渡、不重复、禁止空话
       - 标准：适度停顿、关键点可重复一次
       - 从容：允许铺垫、允许回顾、允许充分互动
   - **强制包含图片可读性约束**：
     * 若用户提供图片：图片是主要信息源、看不清即失败、旁白不得补图外细节
     * 若用户不提供：用示意图/emoji，并避免要求观众读小字
   - **对于类型 C（版本发布）**：
     * 强调技术术语需要配形象化图示（UVC/MQTT/人脸识别等）
     * **图示内容边界硬约束**：示意图/emoji 仅用于解释"概念与场景"，禁止呈现或暗示 CHANGELOG 未给出的 UI 界面、具体命令/输出、具体参数/数值、具体性能提升幅度
     * ✅ 可以用：概念图标（📷摄像头、💬消息、🔐加密）、流程箭头、抽象示意图
     * ❌ 不可以：mock 界面、假命令、编造数据

5. **自检（必须做）**
   - A：是否出现技术原理/代码/参数细节？若是→删/改
   - B：是否"复述文档"而缺少直觉/类比/图示？若是→重写结构
   - C：重要技术术语是否有形象化图示说明？若无→建议补充或用文字解释
   - 是否出现"文档外推测/发明"？若是→删除并标注"文档未覆盖"
   - 是否把"建议截图/素材"写成强制？若是→改为"建议/可选/替代方案"

## Style & Constraints（风格与边界）

- **中文输出**：默认简体中文
- **不做代码级实现细节**：除非用户明确要求"实现/代码"
- **不发明信息**：所有事实必须能在输入文档找到依据
- **渐进披露**：长模板放 `templates/`，主文尽量短、可维护

## Common Mistakes（常见错误 - 避免）

### ❌ 错误 1：编造 CHANGELOG 未提及的细节

```
错误示例（Prompt 中写）：
"MQTT 客户端库支持 QoS 0/1/2 三种服务质量等级，吞吐量提升 50%"

问题：CHANGELOG 只说 "Added MQTT client library with TCP/SSL support"，没有提及 QoS 和性能数据

正确做法：
"MQTT 客户端库：设备之间发送消息的协议，支持 TCP 和 SSL 连接"（只使用原文信息）
```

### ❌ 错误 2：把"建议截图"写成"强制要求"

```
错误示例：
"必须上传以下 5 张截图，否则视频无法生成"

正确做法：
"建议上传以下截图（可选）：
- P0：MQTT 消息传递示意图（替代方案：用 💬 emoji 表示）
- P1：人脸识别效果图（替代方案：用 👤 emoji 表示）"
```

### ❌ 错误 3：用 mock 界面/假命令代替概念图示

```
错误示例（图示内容）：
画一个 "mqtt_connect('tcp://192.168.1.1:1883')" 的代码截图

问题：CHANGELOG 未给出具体 API，这是编造

正确做法：
用抽象的消息传递示意图（设备A → 云 → 设备B）解释"MQTT 是什么"
```

### ❌ 错误 4：逐条念 Fixed 列表

```
错误示例：
"修复了音频覆盖问题，修复了 GPIO 中断问题，修复了 PWM 频率问题，修复了..."（念 20 条）

正确做法：
"修复了音频、GPIO、PWM 等 12 个驱动和组件问题，提升系统稳定性"（按类别概括）
```

### ❌ 错误 5：自己编造"下一版本预告"

```
错误示例：
"下一版本我们计划支持 USB Host 和蓝牙 Mesh..."

问题：CHANGELOG 中没有 Roadmap 章节

正确做法：
"欢迎升级体验 0.1.2 版本，更多功能敬请期待"（不编造预告）
```

## Examples（示例调用）

### 示例 1：产品/商务 Feature（A）

**用户输入**：给你这份 md，请做一个 2 分钟新功能介绍视频，面向产品和销售。  
**你输出**：`assets_checklist.md`（建议）+ `notebooklm_prompt.md`，并确保不出现实现细节。

### 示例 2：技术原理讲解（B）

**用户输入**：把这段架构原理做成教学视频，观众是后端工程师。  
**你输出**：`assets_checklist.md`（建议）+ `notebooklm_prompt.md`（可直接粘贴到 NotebookLM）。

### 示例 3：版本发布视频（C）

**用户输入**：给你这份 CHANGELOG.md，请做一个版本发布视频，面向开发者。  
**你输出**：`assets_checklist.md`（建议补UVC/MQTT等技术术语的形象化图标）+ `notebooklm_prompt.md`（固定三部分结构：新Feature → 问题修复 → 下一版本预告）。
